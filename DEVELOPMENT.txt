# langur development
2024-11-19

I hope you find langur useful. I've put a lot of work into it. It started as a project for learning, not as an end in itself, but I have made drastic improvements since. The syntax is relatively settled at this point, so I'd like to get to 1.0 soon. I only have so much time. If you'd like to contribute to the project, here are some things I'd like to see. While I don't have time for all of it, I do have ideas about how it could function and the syntax. I'm open to input.

To make 1.0, I would like to fix setting by index. There are at least a couple of issues here. One is efficiency. The other is that you cannot presently set by index if using more than a single index on a variable (as shown below).

x[1] = 123      # okay
y[1][2] = 123   # doesn't work

I make note of two kinds of "purity," one in regard to side effects, and one in regard to variables. Langur presently has strict purity in terms of variables. I'm not absolutely settled on this, so long as changes to this can be made that don't make concurrency difficult.

--Anthony Davis

backlog
- value forwarding syntax with one or multiple uses of forwarding variable _fwd
    instead of writing ...
        join(filter(mapX(wordsets..., by=amb)), by="\n")
    one could write ...
        mapX(wordsets..., by=amb) -> filter -> join(_fwd, by="\n")

    If there are multiple uses of the forwarded value after an operator, the compiler could assign it to a temporary internal variable first, so that it is evaluated once.

- compile on AST node instead of seperate compile package
    Work was started toward this, but not finished.
- compile with metadata that can be used in VM to traceback errors to source code
    ditto
- log (math) function
- atan2(), acos(), etc.
- duration string formatting, including weeks
- engineering notation (almost like scientific notation) interpolation modifier
- possible changes to parameters to specify adding/removing trailing zeroes for round() and trunc()
- PEG grammar
- use VCS (github likely)

other things to consider...
- efficiency
    While we couldn't expect performance like C, Go, or Rust, it could be improved.
    Append is way too slow.
- resource management
    close resources when finished, even if there's an exception
- dependency management
- import modules
    presently parsed, but not compiled
    must account for global modes
    maybe move some built-in functions to be imports or pseudo-imports, uncluttering the global namespace
        use more descriptive names, which could be aliased to short names
- module compile
    global modes
    order irrelevant for module compiled declarations
        presently parses module and compiles declarations in reverse order
- structures
- OOP with composition
    no inheritance - I don't like inheritance in programming.
    I also don't like "everything is an object."
- optional explicit typing
- complex numbers
- bitwise operations
    at least on explicit types
- pointers
    using strict left-to-right syntax; none of the left/right business you see in C
- concurrency
    maybe using go/conc library
    must trap/account for exceptions
- testing mechanisms
