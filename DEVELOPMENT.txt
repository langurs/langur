# langur development
2025-04-22

I hope you find langur useful. I've put a lot into it. It started as a project for learning, not as an end in itself, but I have made drastic improvements since. The syntax is relatively settled at this point, so I'd like to get to 1.0. I only have so much time. If you'd like to contribute to the project, here are some things I'd like to see. While I don't have time for all of it, I do have ideas about how it could function and the syntax, and I'm open to input.

To make 1.0, I would like to fix setting by index. One issue is efficiency. Another is that you cannot presently set by index if using more than a single index on a variable (as shown below).

x[1] = 123      # okay
y[1][2] = 123   # doesn't work

--Anthony Davis

backlog
- compile on AST node instead of having a seperate compile package
    Work was started toward this, but not finished.
- compile with metadata that can be used in VM to traceback errors to source code
    Work was started toward this, but not finished.

- write custom decimal library
  make safe for concurrency
  simplify and other changes
- PEG grammar
- log (math) function
- atan2(), acos(), etc.
- duration string formatting, including weeks
- engineering notation (almost like scientific notation) interpolation modifier
- value forwarding syntax with one or multiple uses of forwarding variable _fwd
    instead of writing ...
        join(filter(mapX(wordsets..., by=amb)), by="\n")
    one could write ...
        mapX(wordsets..., by=amb) -> filter -> join(_fwd, by="\n")

    If there are multiple uses of the forwarded value after an operator, the compiler could assign it to a temporary internal variable first, so that it is evaluated once.

other things to consider...

    maybe move some built-in functions to be imports or pseudo-imports, uncluttering the global namespace
        use more descriptive names, which a programmer could alias to short names if he wants to

    global modes
    order irrelevant for module compiled declarations
